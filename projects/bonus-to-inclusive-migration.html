Bonusleistungen → Inklusivleistungen Migration

A catalogue-wide data analysis that turned a "migrate everything" instinct into a three-way routing decision — so 29 standardised perks landed in the right place in the new data model, with dedup logic to prevent thousands of offers gaining duplicate entries.

Data analysis Migration B2B · Hospitality Shipped

The problem

The legacy system stored two overlapping kinds of offer text. Bonusleistungen were standardised dropdown perks — around 29 predefined options like Sauna, Welcome Drink, WLAN, Parkplatz. Inklusivleistungen were free-text "what's included" lines, written by hand per offer. In practice, hotels used both: they'd pick Sauna from the dropdown and also write "Sauna" into the free text.

Migrating the dropdown perks into the new KMW inclusives model naïvely would create duplicate entries on thousands of offers — "Sauna" as a migrated inclusive and "Sauna" already in the free text sitting alongside it. It would also bring the long tail of rarely-used perks into the new system, where they'd clutter the data model without adding value.

The analysis

I worked through the full offer dataset (~22 MB export, all offers in the catalogue) and did two things:

First, semantic overlap detection: for each of the 29 Bonusleistungen, I identified where the same perk was already written into the free-text Inklusivleistungen on the same offer — to understand where a naive migration would create duplicates, and what keywords would reliably catch it.

Second, per-perk usage analysis: how many offers actually used each Bonusleistung across the whole catalogue. Usage ranged from around 3.6% for common perks down to well under 1% for the long tail — which made the case for dropping the low-value ones rather than carrying them forward.

The mapping decision

The analysis turned into a three-way routing for all 29 perks:

Action Count Rule
→ Inclusive 16 Migrate as an ArrangementInclusive, each with dedup keywords so it's skipped when the offer's free text already mentions it
→ Hotel attribute 2 WLAN and Parkplatz become hotel-level attributes, not per-offer inclusives — they're hotel-wide facts, not offer-specific perks
→ Drop 11 Dropped — under 4% catalogue usage, or obsolete (e.g. public internet terminal). Each annotated with the reason

The WLAN/Parkplatz decision was the one that required the most explanation internally: hotels had been entering these as per-offer perks, but they're not properties of an offer — they're properties of the hotel. Moving them to the hotel attribute layer meant the data ended up in the right place, even though it changed where hoteliers would manage them going forward.

Dedup logic

The 16 perks that migrated each came with a set of dedup keywords. The persistence logic checks these at migration time: a new inclusive is only added if none of its keywords already appear in that offer's existing free-text inclusives. If they do, the migrated entry is skipped.

Example — Wellness perk
Perk Wellness
Dedup keywords Wellness · Vital · Erholung · Relax · Schwimmbad
Result If any of these already appear in the offer's Inklusivleistungen, the migrated "Wellness" inclusive is skipped — preventing duplication at scale

Choosing the right keyword set for each perk was the detail work: broad enough to catch real overlaps, specific enough not to skip perks where the overlap is only superficial.

Outcome and my role

The mapping decision shipped as part of the platform migration (ticket HEX-1049), implemented by engineering and merged to the default branch. The outcome: a clean migration of standardised perks into the new model, without duplicating existing free-text inclusives and without carrying the low-value long tail — driven by usage data rather than migrating blindly.

My contribution was the analysis and the mapping decision: which perks migrate, which become hotel attributes, which are dropped, and the dedup-keyword strategy for each. The code was written by engineering — this is a data/product piece, not code I shipped.
