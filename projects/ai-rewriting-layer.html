AI Rewriting Layer — Duplicate-Content Avoidance

After a platform migration left two consumer portals serving identical offer texts, Google started treating them as duplicate content. I designed the fix and built the founding module — a compound AI backend service that automatically rewrites content to make it unique, with guard layers to prevent hallucination and a swappable provider layer so the model is a config change, not a code change. Working prototype on a feature branch; a developer later extended it.

The problem

After the OneExtranet migration, two consumer portals — KMW and KU — were both serving the same offer texts: identical titles, identical descriptions, same phrasing. Google treats this as duplicate content, which creates an SEO risk across both portals and puts search-driven revenue at stake.

The fix: an automated rewriting layer that generates unique text for KU, keeping KMW as the master and KU as the AI-rewritten version. The requirement wasn't to produce better text — it was to produce different text that preserved all the original facts, respected SEO constraints, and was safe enough to go live without a human reviewing every line.

What I built

A compound AI backend service in PHP/Symfony — not a script that calls an LLM. The founding module rewrites offer texts (title + intro paragraph) for the KU portal. A developer subsequently extended the branch with a GraphQL API, admin review UI, and Twig-templated prompts; the architecture below covers my commits.

1
Context enrichment
Hotel name, city, region, and detected amenities injected into the prompt before generation.
Shipped
2
Generation — swappable LLM provider
One interface, three providers (OpenAI, Anthropic, Gemini) — chosen by config. Model changes don't touch the pipeline. SEO- and brand-tuned German system prompt: title ≤35 chars with location, intro 200–500 chars with key keywords in the first 155, KU tone of voice, no prices, no invented features.
Shipped
3
Rule-based validation + self-correction loop
Length limits, required fields, hotel-name-not-in-title. If validation fails, the errors are fed back to the model as a correction prompt and it retries once — temperature drops from 0.7 to 0.3 on the second attempt.
Shipped
4
Two guard layers
Content guard: term blocklist + OpenAI moderation API. Semantic/fact-check guard: flags any amenity (pool, sauna, spa…) that appears in the generated text but not in the source data — plus language, price, and star-rating checks.
Shipped
5
Idempotent persistence
Results stored per (arrangement, portal) with a SHA-256 source hash and a prompt version. Unchanged offers are skipped on re-runs; a prompt version bump triggers a full re-run. Status tracked as Pending / Generated / Failed / Approved / Rejected.
Shipped
6
Human review queue + KU sync
Flagging high-value hotels for editorial review before the MasterData API syncs to KU — plus a CVR dashboard comparing AI vs. human text performance. Not yet built.
Concept

The batch CLI runs via app:arrangement:rewrite with options for batch size, dry-run, retry-failed, and single-arrangement targeting. The whole service is behind an AI_REWRITING_ENABLED feature flag. Full unit test suite across the orchestration service, both guards, the validator, and the provider factory.

Evidence it works

I ran the rewriter across 14 hotels and 161 offers, comparing OpenAI and Gemini output side by side. Two examples:

Hotelferienanlage Friedrichsbrunn (Harz) — offer #70890
Original "All inklusiv kurz und unvergesslich – 4 Tage im Harz"
OpenAI "4 Tage Wellness in Thale" — location moved to front, duration kept
Gemini "Thale: 4 Tage All-Inclusive Genuss"
Mövenpick Hotel Hamburg City — offer #782952
Original "KARNEVAL SPECIAL: Kurztrip nach Hamburg | 3 Tage"
OpenAI "3 Tage Hamburg: Entdecke die Stadt"
Gemini "Hamburg: 3 Tage Städtereise-Special"

Both rewrites keep duration and city, drop verbatim phrasing from the original, and lead with the location keyword — the duplicate-content goal, met. The intro paragraphs also moved location to the front for SEO, and the amenities named in the output (sauna, pool) were ones present in the source data — the hallucination guard doing its job.

My role

I owned the concept — problem framing, business case, the 6-phase compound AI architecture, and the cross-cutting concerns (semantic cache, human review queue, measurement, EU AI Act / GDPR-AVV implications). I then built the founding backend module myself: the initial commit that created the service, the provider layer, both guards, the validator, the entity + migration, the CLI command, and the unit test suite. I also owned the prompt and guard design — the SEO rules, the no-prices / no-invented-features constraints, the hallucination check logic — and ran the provider A/B evaluation on real data.

A developer subsequently extended the branch: a GraphQL API and admin comparison view, externalising prompts into Twig templates with content-hash versioning, prompt-injection/XSS prevention, a service refactor, and an approve/reject review workflow. Of the commits on the branch, the founding module and prompt work are mine; the hardening and UI extension are the developer's.

Status: working prototype on a feature branch. Not merged to main or deployed to production. The broader program (hotel descriptions first for higher SEO impact, the review queue, the CVR dashboard) builds on this foundation.
